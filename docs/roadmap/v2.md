# Feather v2 — "It survives the scary sites, safely"

> Front door: [`feather.md`](../../feather.md). Sibling roadmaps: [v1](v1.md), [v3](v3.md).
> Execution index (phases/sessions/ADRs): [`ROADMAP.md`](../../ROADMAP.md).

**Goal:** an agent works on locked-down, bot-detecting sites (LinkedIn, Instagram, insurance portals)
*as you*, with your approval, without getting blocked. This is the **Cookie Mine paying off** — and
it's where Feather's security model lives.

**Security first.** v2 builds the safety machinery *before* any agent acts in a real warmed profile.
(From the 2026-06-07 five-model council review: Feather had been sequencing its highest-privilege
surfaces before the safety that governs them — reversed here.)

---

## Build order (the security-first spine)

`capability/safety gate → Identity → MFA → warmed-profile attach → Stealth hardening (last)`

Two decisions make this order possible:
1. A **session-hold primitive** replaces MFA's direct stealth toggle → frees **Stealth to be last**
   (it's the most complex/breakable).
2. **Identity** stores stealth/MFA policy as opaque/versioned refs, not hard imports → frees
   **Identity to go first**.

### Security gates (do not skip)
- **Gate A — capability gate:** must exist before warmed-profile attach, MFA local routes, or vault
  activation. (Session 5.0.0, implements [`adr-0010`](../specs/adr-0010-local-control-plane-capability-model.md).)
- **Gate B — first-agent safety gate:** credential-at-rest posture + leakage harness before any agent
  acts in a *warmed* profile. (Session 5.0.2.)

## Features

- [ ] **Safety / capability gate (control plane)** — Session 5.0.0 (implements ADR-0010)
- [ ] **Identity Model** — name your warm profiles so an agent attaches "as your Gmail," pre-logged-in.
      Plan: [`docs/specs/2026-06-07-identity-model-plan.md`](../specs/2026-06-07-identity-model-plan.md)
- [ ] **MFA Handler** — login-wall → phone notification → you approve → agent resumes (no raw codes to
      the agent). Plan: [`docs/specs/2026-06-07-mfa-handler-plan.md`](../specs/2026-06-07-mfa-handler-plan.md)
- [ ] **Stealth hardening (full)** — always-on consistency checks, secure-by-default.
      Plan: [`docs/specs/2026-06-07-stealth-stack-plan.md`](../specs/2026-06-07-stealth-stack-plan.md)
- [ ] **Learn-your-behavior** — observe how *you* use the mouse/keyboard and teach the agent your
      human signature (kinematic input; spike-first, gated on findings)
- [ ] **Active anti-bot self-detection** — watches its own behavior live and corrects when it trends
      robotic (the active version of v1's read-only self-check)
- [ ] **Teach-a-workflow / action cache** — show a task once, replay it forever without the LLM
      (Anchor-inspired determinism layer)
- [ ] **MCP tool-surface reconciliation** — Session 5.0.1

## The first test

Create Feather's **own LinkedIn account** (collaborative, human-in-the-loop) and operate it
un-flagged — the harder cousin of v1's Instagram test, and the proof that the stealth hardening works.

## Recipe books for v2

- **fingerprint npm** (`fingerprint-generator` / `-injector`) — **buy *if*** hardening needs spoofing.
  Note: the current stealth plan deliberately **verifies rather than spoofs** (a real browser already
  has a genuine fingerprint and detectors look for *tampering* first), so these may not be needed.
- **Maxun** workflow DSL — reference only (AGPL); port the `where/what` *idea* by hand for
  teach-a-workflow.
