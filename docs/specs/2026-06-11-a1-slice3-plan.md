## Goal

Finish Gate A as *proven* machinery, not bench-tested machinery: the human approval surface, the
off-by-default dangerous-mode policy, the durable audit log, revocation wired to real session close —
and **one real Dangerous-tier door (cookie export)** so the full
`request → human approve → use → audit → revoke` loop runs end-to-end against the scratch profile.

## Joint calls locked at the 2026-06-11 brainstorm

1. **Approval ping = terminal + event stream.** The approval URL is printed to the Feather server
   console and a redacted `grant.requested` event goes on the SSE bus. No desktop/Telegram
   integration now.
2. **Grants die on restart.** Registry stays in-memory; restart = fail-closed (re-request +
   re-approve). Only the **audit log** persists.
3. **TTL = 60s flat** for all capabilities; per-capability tuning deferred until real ops exist.
4. **Scope = gate + one demo door (cookie export).** Roi wants to live-test with `scratch`. CDP
   attach (5c) and vault (frozen) get gated when built.

## Security invariants (from ADR-0010 / MFA plan — non-negotiable)

- **The agent never holds a grant secret.** The agent-facing request route returns
  `{grantId, status}` only — no approval URL, no `humanToken`, no nonce. The approval URL
  (carrying a single-use 256-bit CSPRNG `humanToken`) is surfaced via the server console + the
  human's channel only. SSE events carry ids/status, never tokens.
- **Use is consume-by-scope, not present-a-secret:** the export endpoint asks the registry to
  consume the granted record for `(sessionId, "cookie-export")`. The slice-2 nonce stays as the
  internal single-use linkage; nothing external transports it.
- **Approval page = the MFA S1 stack:** `humanToken` in URL (single-use), per-page CSRF nonce
  posted back, strict CSP (no external resources), and the global A0 Origin/Host guard already in
  front of everything.
- **Dangerous capabilities are off by default** (`FEATHER_DANGEROUS_CAPABILITIES` env, comma list,
  e.g. `cookie-export`). Request route refuses (403 `DANGEROUS_DISABLED`) when not enabled.
- **Audit both surfaces:** every `grant.*` lifecycle event → SSE bus + append-only JSONL under the
  STATE root (`<state>/logs/audit/grants.jsonl`). Redacted (records only — registry events already
  carry no nonce).

## Tasks (TDD per behavior, in order)

1. **`consumeGranted(sessionId, capability)` on `CapabilityGrantRegistry`** — finds the granted,
   unexpired record for the scope and redeems it internally; misses return the precise reason
   (`no-grant` / `expired` / wrong state). Unit tests.
2. **Dangerous-mode policy** (`src/capability/policy.ts`) — parse `FEATHER_DANGEROUS_CAPABILITIES`,
   `isEnabled(capability)`. Default: nothing enabled. Unit tests.
3. **Durable audit sink** — `onEvent` → append JSONL line `{ts, type, grant}` to
   `<state>/logs/audit/grants.jsonl` (new `FeatherPaths` helper + init dir) **and** SSE bus emit.
   Unit test with temp dir.
4. **Approval store + page + routes** (`src/capability/approval.ts` + transport wiring):
   - In-memory `humanToken → {grantId, csrfNonce}` store; 256-bit CSPRNG; single-use.
   - `GET /v1/approvals/:humanToken` (no-auth, local page): shows session/capability/TTL,
     Approve + Deny buttons, embedded CSRF nonce, strict CSP headers. Reuses the resume-page
     rendering style.
   - `POST /v1/approvals/:humanToken` `{action, csrf}` → approve/deny in registry; consume token;
     confirmation/expired pages.
   - Unit + integration tests incl.: foreign-Origin POST rejected (A0), wrong/reused
     `humanToken` rejected, wrong CSRF rejected.
5. **Agent-facing request route** — `POST /v1/sessions/:id/grants` `{capability, ttlMs?}` (authed):
   policy check → `registry.request()` → mint humanToken → console-print approval URL + SSE
   `grant.requested` → return `{grant}` (no URL/token). Integration test asserts the response
   carries no token/URL.
6. **Session-close wiring** — server owns one `SessionHoldRegistry` + one `CapabilityGrantRegistry`;
   session close path calls `releaseAllForSession` + `revokeAllForSession`. Integration test:
   approve a grant, close the session, export refuses with `revoked`.
7. **The door: `POST /v1/sessions/:id/cookies/export`** (authed) — policy check →
   `consumeGranted(id, "cookie-export")` → `context.cookies()` → respond `{cookies}` + audit.
   No grant → 403 `GRANT_REQUIRED`. Integration: full happy loop + deny loop + expiry loop.
8. **Docs** — `docs/api-reference.md` (grants + approvals + export), playbook note. Gate A design
   doc gets a one-line "slice 3 shipped" status touch.

**Out of scope:** CDP-attach/vault wiring (5c / ADR-0008), MFA challenge auto-revoke trigger (lands
with 5b — the registry API for it exists), notifier abstraction beyond console (MFA plan owns it),
grant persistence (decided against), policy layer observing holds (later A1/5b work).

**Live proof (after implementation, with Roi):** scratch session → request export → approve on the
local page → cookies land → then the honest failure runs: deny, expire (60s wait), revoke-by-close.
