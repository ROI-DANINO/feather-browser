# Session 5.0.2 — First-Agent Safety Gate

**Phase:** 5.0 — Agent Interface And Safety Gate
**Status:** Future.

## Goal

Close the safety blockers before any agent acts inside a human-warmed profile. Complements the
capability gate (5.0.0) with the *credential-at-rest* and *leakage* prerequisites the council called
out.

## Tasks

- Measure `primary` DBSC binding read-only before any clone/copy decision.
- Ensure warm profiles disable Chromium's built-in password manager.
- **Treat the warmed profile as credential-at-rest** (council): restrictive FS permissions, no
  world-readable / cloud-synced profile paths, per-identity deletion, and at-rest encryption
  (LUKS / encrypted home) or OS keyring via DBus (libsecret / GNOME Keyring / KWallet) with
  `vaultRef` as a **keyring locator, not a JSON path**.
- Re-run secret-leakage harnesses before credential-related work.
- Decide whether vault Spikes A/B are required before this phase continues.

## Expected output

- A documented "first agent action" gate, including credential-at-rest posture.

## Definition of done

- No agent can accidentally receive raw credentials or corrupt the primary trust jar.
- The warmed-profile-at-rest prerequisites are met or explicitly waived with rationale.

## Risks / blockers

- `primary` may be device-bound; never clone it blindly.
- Vault ADR-0008 remains candidate until SQLCipher/KeePassXC spikes clear.

## Do not do yet

- Do not implement vault credential release.
- Do not copy real primary cookies.
