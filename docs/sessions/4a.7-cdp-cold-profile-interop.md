# Session 4a.7 — CDP/WS Endpoint Exposure (Cold-Profile Interop Proof)

**Phase:** 4a — Feather Core Open-Source Readiness And Public Proof
**Status:** Pending — recommended next implementation session.
**Classification:** Core interop quick win — **scoped to cold/throwaway profiles only.**

> **Re-scoped by the 2026-06-07 council review (5/5) and Session 4a.6b.** CDP is effectively root
> access to a session (`Network.getCookies` / `Storage.getCookies` / `Runtime.evaluate` read raw
> tokens). Exposing it on a **warmed** profile before the Phase 5.0 capability gate contradicts the
> core security thesis. **Decision (4a.6b): ship only the cold/throwaway-profile interop proof now;
> defer warmed-profile attach to Phase 5c (warmed CDP attach), behind the ADR-0010 capability gate.**
> See `research/2026-06-07-council-design-review.md` and `docs/specs/adr-0010-local-control-plane-capability-model.md`.

## Goal

Expose the browser CDP / Playwright WebSocket endpoint in the launch/session response **for
cold/disposable profiles**, so Browser Use and Crawl4AI-style tools can attach to a
Feather-managed *throwaway* session. Prove the interop mechanism without exposing any warmed
credentials.

## Tasks

- Verify the current Playwright 1.60 CDP/WS endpoint shape from official docs before coding.
- Identify whether Feather already holds the endpoint and where it should be stored.
- Add a stable, sanitized endpoint field to `SessionRecord` / launch output **only for
  cold/disposable profiles**. For persistent/warmed profiles the field is **absent** (not empty) and
  the response notes that warmed attach requires the Phase 5.0 capability grant.
- Keep the endpoint behind the existing token auth and loopback bind; do not widen remote exposure.
- Add focused unit/integration coverage, including a test that asserts the endpoint is **not**
  exposed for a persistent profile.
- Document "attach Browser Use / Crawl4AI to a Feather cold session" with a minimal example —
  **without importing** Browser Use or Crawl4AI.

## Expected output

- Launch/session responses expose an attachable endpoint **for cold sessions only**.
- Cold-profile interop is documented; warmed attach is documented as "deferred behind the Phase 5.0
  capability gate."

## Definition of done

- Tests and typecheck pass, including the "no endpoint on warmed profile" assertion.
- API docs explain the field, its cold-only availability, and the warmed-attach deferral.
- No auth-token leakage and no extra remote exposure.

## Risks / blockers

- Dual CDP clients can conflict; document this honestly.
- Endpoint exposure must not accidentally widen access beyond loopback/token-authenticated callers.

## Do not do yet

- **Do not expose CDP on warmed/persistent profiles** — that is Phase 5c, behind ADR-0010.
- Do not build MCP or an agent framework.
- Do not import Browser Use or Crawl4AI.
