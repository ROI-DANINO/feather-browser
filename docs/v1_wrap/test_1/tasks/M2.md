  https://httpbin.org/forms/post returned HTTP 503 with the page "503 Service Temporarily

- EVIDENCE: Feather `navigate` returned `status: 503` on 4 consecutive attempts. `observe` returned

  `snapshot` confirmed the body was literally "# 503 Service Temporarily Unavailable". I did NOT
  assume failure or success — I read the actual rendered page.
- LESSON: The block is browser-specific, not a Feather targeting/timing issue. Running
  `curl https://httpbin.org/forms/post` in parallel returned HTTP 200 every time, even with a spoofed
  Chrome user-agent — so plain HTTP clients get served but the real Chromium request gets 503.
  httpbin.org's hosting/CDN is throttling or bot-blocking the full browser request fingerprint
  (TLS/header ordering, not just UA, since UA spoofing on curl still passed). Things to try next time:
  a warmed/persistent profile instead of disposable, routing through a residential proxy,
  slowing/retrying with backoff over a longer window, or using an alternative echo endpoint (e.g. a
  self-hosted httpbin or postman-echo) to validate the fill→submit→verify loop. The Feather drive
  loop itself worked correctly end to end — observe accurately reported the empty 503 page rather than
  hallucinating a form.
